One pod, any GGUF model, one API.

A self-hosted RunPod script that serves quantized GGUF checkpoints behind an OpenAI-compatible endpoint, fully controlled from a single Gradio panel. Models live on the network volume only — the container filesystem is never the source of truth.

Key decisions

  • ✓ DECIDED Build on llama-server as the inference + API process. No custom OpenAI-compat layer gets written from scratch.
  • ✓ DECIDED All weights resolve to the network volume via HF_HOME / local_dir. The container's default HF cache is never used.
  • ✓ DECIDED Gemma 4 12B is a standard dense multimodal model — it is not diffusion-based — and is in scope from day one.
  • ⏳ BLOCKED DiffusionGemma (26B-A4B, block-diffusion) is excluded from MVP. Mainline llama.cpp support is not merged (tracking PR #24427).
  • ✓ DECIDED Pod, not Serverless. On RTX 6000 Ada 48GB: Pod ≈ $0.77/hr vs Serverless ≈ $1.90/hr — roughly 2.5× markup, and serverless cold starts get worse with multi-GB GGUF loads. Gradio also needs a long-lived process regardless. Stop the pod manually when idle; GPU billing stops with it, only network-volume storage keeps accruing.
  • ✓ DECIDED One active model in llama-server at a time, switched on demand. The 48GB card has headroom for 2-3 concurrent models, but that's deliberately deferred — see Roadmap.

System

Control plane and data plane, kept separate

Gradio owns nothing about inference itself — it just tells a process manager which model to run. The process manager owns one llama-server child process at a time.

Storage

If it's not on the volume, it doesn't exist

Every download path is pinned to the network volume before any Hugging Face call happens. The container's local disk is treated as ephemeral and is never relied on for model weights.

Cost of keeping 2-3 models around

Network Volume runs ~$0.07/GB/month, not Container Volume (which bills 2× while a pod is stopped). Three 12B models at Q5_K_M/Q8_0 quants land around 25-45 GB total — roughly $1.5-3/month. Storage is noise next to the GPU bill; no need to quantize harder just to save disk space.

Download mechanics

Use hf_hub_download(..., local_dir=<volume path>, allow_patterns=[chosen_file]) rather than cloning a full repo — most GGUF repos contain every quant, and only one is needed per download. A free-space check against the volume runs before the transfer starts, not after.

Lifecycle

Download, quantize-aware, then switch

Two flows: pulling a new model onto the volume, and pointing llama-server at a model that's already there.

Download flow

  1. Paste a HF repo id — plus an access token for gated or private repos. Token is stored as a secret, never echoed back into the UI.
  2. List available files via the HF API, grouped by quant (Q4_K_M, Q5_K_M, Q8_0, …) with size shown against the ladder from Overview.
  3. Pick a quant — UI checks it against free volume space before allowing the download to start.
  4. Download straight to the volume with a progress bar; no container-local intermediate copy.
  5. Auto-detect an mmproj file in the same repo (vision/audio models like Gemma 4 12B) and offer to pull it alongside.

Switch flow

  1. Pick a model from "Models on volume" — already-downloaded files only, no network call.
  2. Process manager stops the current llama-server cleanly.
  3. Process manager starts a new one with the selected file (+ mmproj if present).
  4. active_model.json is written so the same model reloads automatically after a pod restart.

Per-model run config

Kept editable per model rather than hard-coded: context length, -ngl (GPU layers offloaded), batch size, and chat-template override for repos with non-standard metadata.

API

llama-server already speaks OpenAI

No custom API layer gets built. llama-server ships an OpenAI-compatible surface — Gradio is a control panel in front of it, not a proxy.

Auth & exposure

RunPod pod ports are reachable publicly by default, so --api-key is passed to llama-server from the first MVP build, not added later. Every request requires a Bearer token. Gradio itself talks to llama-server over localhost — only the API port needs to be exposed if the goal is pure API consumption from outside the pod.

Model compatibility

Check the architecture before the download, not after

"Unknown model architecture" only surfaces at load time — after a potentially multi-gigabyte download. Validate against llama.cpp's supported-architecture list first.

Build order

MVP first, then resilience, then concurrency

MVP

Ships in the first working build

  • HF download to volume, with token support
  • Single active llama-server, OpenAI-compatible API
  • Gradio: library, switch, console
  • active_model.json persistence across restarts
Phase 2

Resilience and safety

  • Auto health-check + restart on crash
  • Disk-space guard + quant size preview before download
  • mmproj auto-pairing for multimodal repos
  • API key auth enforced by default
  • Multi-GPU tensor-split config
Later

Concurrency & experimental

  • Multiple models on separate ports behind a small router
  • DiffusionGemma support once PR #24427 lands in mainline
  • Request queue / concurrency-limit UI

Open questions

What could go wrong, on purpose

Exposure

Public port. RunPod ports are reachable by default — the API key requirement is part of the MVP, not a follow-up task.

Concurrency

Single-model constraint. Switching kills in-flight requests. MVP needs at least a clear warning before switching; a "draining" state is a Phase 2 candidate.

Upstream

DiffusionGemma timeline is entirely outside this project's control — tied to llama.cpp PR #24427 merging.

Disk

Volume sizing. Keeping F16 plus several quants of the same model fills a volume fast. Worth a retention or cleanup policy once more than a couple of models are in rotation.

Secrets

HF token handling. Must never land in logs, in active_model.json, or in any other file written to the volume.
